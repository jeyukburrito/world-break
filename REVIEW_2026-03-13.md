## 수정 필요

### 1. [HIGH] `users` INSERT RLS 정책 누락

- **파일**: `supabase/rls.sql`
- **문제**: `users` 테이블에 SELECT/UPDATE 정책만 존재. INSERT 정책이 없어서 Google 로그인 후 User 레코드 생성 시 RLS에 의해 차단됨.
- **해결**: 아래 정책 추가

```sql
create policy "users_insert_own"
  on public.users
  for insert
  with check (auth.uid() = id);
```

또는 Supabase `auth.users` → `public.users` 동기화 트리거를 사용하는 경우 service_role로 처리.

### 2. [MID] 폰트 미로드

- **파일**: `app/globals.css`, `app/layout.tsx`
- **문제**: `globals.css`에서 `"Noto Sans KR"` 지정했지만 폰트 로드 코드 없음. 시스템 sans-serif로 폴백됨.
- **해결**: `app/layout.tsx`에서 `next/font/google` 사용

```tsx
import { Noto_Sans_KR } from "next/font/google";

const notoSansKr = Noto_Sans_KR({
  subsets: ["latin"],
  weight: ["400", "600", "700"],
});
// <body className={notoSansKr.className}>
```

### 3. [MID] `/matches/new` 내 덱 필드 타입 불일치

- **파일**: `app/matches/new/page.tsx`
- **문제**: "내 덱" 필드가 `<input type="text">`이지만, Prisma 스키마에서 `myDeckId`는 `Deck` 테이블 FK (UUID). 유저의 등록 덱 목록에서 선택하는 `<select>`여야 함.
- **해결**: Phase 2에서 서버 컴포넌트로 유저 덱 목록을 조회하고 `<select>` 또는 커스텀 드롭다운으로 교체.

### 4. [LOW] `.env.example`과 `.env.local.example` 중복

- **파일**: `.env.example`, `.env.local.example`
- **문제**: 두 파일이 동일 내용. `SUPABASE_SETUP.md`에서 `.env`(Prisma CLI)와 `.env.local`(Next.js)을 분리하라고 안내하지만, 예제 파일이 같으면 혼동 가능.
- **해결 옵션**:
  - A) `.env.example`은 `DATABASE_URL`과 `DIRECT_URL`만 남기고, `.env.local.example`에 전체 키 유지
  - B) `.env.local.example` 하나로 통합 + README에 "Prisma CLI용으로 `.env`에도 DB URL 복사" 안내

### 5. [LOW] AppShell 헤더 인증 상태 미반영

- **파일**: `components/app-shell.tsx`
- **문제**: 헤더 우측에 항상 "로그인" 링크 표시. 인증된 상태에서도 동일.
- **해결**: Phase 2에서 `AppShell`에 user prop을 전달하여 로그인/로그아웃 분기.

## 구현 시 주의사항

### `isMatchWin` 자동 계산

- `prisma/schema.prisma`의 `MatchResult.isMatchWin` 필드는 Zod 스키마(`lib/validation/match.ts`)에 포함되지 않음 — 이는 올바른 설계.
- 서버 액션에서 match 생성 시 `isMatchWin: wins > losses`로 자동 계산할 것.

### Deck 삭제 시 `onDelete: Restrict`

- `MatchResult.myDeck` 관계가 `onDelete: Restrict`이므로 매치 기록이 있는 덱은 삭제 불가.
- UI에서 "이 덱에 연결된 기록이 있어 삭제할 수 없습니다" 안내 필요.
- 또는 `isActive: false`로 비활성화만 허용하는 soft delete 패턴 권장 (스키마에 `isActive` 이미 존재).
